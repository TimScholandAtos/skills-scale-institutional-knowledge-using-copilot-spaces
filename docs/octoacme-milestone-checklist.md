## Purpose
Provide phase-specific collaboration guidance to ensure effective cross-role coordination at each project milestone.

## How to Use
- Reference at the start of each project phase
- Use as a meeting preparation checklist
- Adapt based on your project's specific needs
- Check off items as completed to track progress

---

## Milestone 1: Project Initiation

### Kickoff Meeting Preparation
**Who's Involved:** Project Manager (lead), Product Manager, Business Analyst, key stakeholders

**Pre-Meeting:**
- [ ] **PM:** Schedule kickoff meeting with all key stakeholders
- [ ] **PM:** Share draft project charter for review
- [ ] **PdM:** Prepare vision and success metrics presentation
- [ ] **BA:** Gather initial stakeholder requirements and pain points
- [ ] **PM:** Prepare meeting agenda and objectives

**During Meeting:**
- [ ] **PdM:** Present project vision, objectives, and success criteria
- [ ] **BA:** Review known requirements and open questions
- [ ] **PM:** Introduce team members and define roles
- [ ] **PM:** Review high-level timeline and key milestones
- [ ] **All:** Discuss constraints, dependencies, and risks
- [ ] **PM:** Define communication cadence and decision-making process

**Post-Meeting:**
- [ ] **PM:** Distribute meeting notes and action items
- [ ] **PM:** Finalize and share project charter
- [ ] **PM:** Set up project board and communication channels
- [ ] **BA:** Schedule requirements workshops

---

## Milestone 2: Requirements & Design

### Requirements Gathering
**Who's Involved:** Business Analyst (lead), Product Manager, UX Designer, Developers, Security Champion

**Activities:**
- [ ] **BA:** Conduct stakeholder interviews and workshops
- [ ] **BA:** Document functional and non-functional requirements
- [ ] **UX:** Conduct user research and create personas
- [ ] **Security Champion:** Review security and compliance requirements
- [ ] **PdM:** Prioritize requirements with stakeholders
- [ ] **BA:** Create initial user stories with acceptance criteria
- [ ] **Dev:** Review requirements for technical feasibility
- [ ] **PM:** Track requirement changes and decisions

### Design Phase
**Who's Involved:** UX Designer (lead), Product Manager, Business Analyst, Developers

**Activities:**
- [ ] **UX:** Create wireframes and initial design concepts
- [ ] **UX:** Review designs with Product Manager and stakeholders
- [ ] **BA:** Validate designs against requirements
- [ ] **Dev:** Review designs for implementation feasibility
- [ ] **UX:** Create high-fidelity mockups and prototypes
- [ ] **UX:** Conduct usability testing with target users
- [ ] **UX:** Incorporate feedback and finalize designs
- [ ] **UX:** Document design specifications for development
- [ ] **PM:** Update project plan based on design scope

**Collaboration Checkpoint:**
- [ ] Design review meeting with all stakeholders
- [ ] Sign-off from Product Manager on designs
- [ ] Confirmation from Developers on technical approach

---

## Milestone 3: Development Planning

### Sprint/Iteration Planning
**Who's Involved:** Project Manager (lead), Product Manager, Developers, Business Analyst, QA, UX Designer

**Pre-Planning:**
- [ ] **BA:** Refine user stories and ensure clarity
- [ ] **PdM:** Prioritize backlog items
- [ ] **Dev:** Review stories for technical dependencies
- [ ] **QA:** Review acceptance criteria for testability
- [ ] **PM:** Prepare sprint/iteration planning agenda

**Planning Session:**
- [ ] **PdM:** Present sprint goals and priorities
- [ ] **BA:** Walk through user stories and acceptance criteria
- [ ] **Dev:** Estimate effort and identify technical dependencies
- [ ] **QA:** Clarify testing approach and requirements
- [ ] **Security Champion:** Flag security considerations
- [ ] **Dev:** Commit to sprint scope based on capacity
- [ ] **PM:** Document sprint backlog and dependencies

**Post-Planning:**
- [ ] **PM:** Update project board with sprint items
- [ ] **Dev:** Create technical tasks and spike stories
- [ ] **QA:** Begin test plan creation
- [ ] **PM:** Communicate sprint goals to stakeholders

---

## Milestone 4: Development & Testing

### Daily Coordination
**Who's Involved:** All team members

**Daily Standup:**
- [ ] **All:** Share progress since last standup
- [ ] **All:** Identify today's focus
- [ ] **All:** Raise blockers or dependencies
- [ ] **PM:** Note and triage any impediments
- [ ] **PM:** Coordinate follow-ups outside standup

### Development Activities
**Who's Involved:** Developers (lead), Business Analyst, UX Designer, Security Champion

**During Development:**
- [ ] **Dev:** Implement features according to acceptance criteria
- [ ] **BA:** Available for requirement clarifications
- [ ] **UX:** Review implementation for design fidelity
- [ ] **Dev:** Write unit and integration tests
- [ ] **Security Champion:** Conduct security code reviews
- [ ] **Dev:** Submit PRs with clear descriptions
- [ ] **Dev:** Address code review feedback
- [ ] **PM:** Track progress and update status

### Quality Assurance
**Who's Involved:** QA (lead), Developers, Business Analyst, UX Designer

**Testing Activities:**
- [ ] **QA:** Execute functional test cases
- [ ] **QA:** Verify acceptance criteria are met
- [ ] **BA:** Validate business logic and workflows
- [ ] **UX:** Conduct usability validation
- [ ] **QA:** Document and track defects
- [ ] **Dev:** Fix critical and high-priority bugs
- [ ] **QA:** Perform regression testing
- [ ] **Security Champion:** Review security scan results
- [ ] **QA:** Sign off on feature completion

**Collaboration Checkpoint:**
- [ ] Mid-sprint check-in with PM and PdM
- [ ] Demo working features to stakeholders
- [ ] Review and adjust scope if needed

## Milestone 5: Pre-Release Preparation

### Release Planning
**Who's Involved:** Release Manager (lead), Project Manager, Developers, QA, Support Lead, Security Champion

**Release Preparation:**
- [ ] **Rel Mgr:** Schedule release window and deployment
- [ ] **Rel Mgr:** Coordinate with dependent teams
- [ ] **QA:** Complete final testing and sign-off
- [ ] **Security Champion:** Verify security requirements met
- [ ] **Dev:** Ensure all code is merged and documented
- [ ] **Rel Mgr:** Prepare release notes and documentation
- [ ] **Support Lead:** Review release changes and prepare support materials
- [ ] **Rel Mgr:** Document rollback procedures
- [ ] **PM:** Communicate release plan to stakeholders

### Pre-Deployment Checklist
- [ ] **QA:** All tests passing in staging environment
- [ ] **Security Champion:** Security scan clean or exceptions documented
- [ ] **Rel Mgr:** Deployment runbook reviewed and validated
- [ ] **Dev:** Database migrations tested (if applicable)
- [ ] **Rel Mgr:** Monitoring and alerts configured
- [ ] **Support Lead:** Support team trained on new features
- [ ] **Rel Mgr:** Stakeholder communication prepared
- [ ] **PM:** Go/no-go decision meeting held

---

## Milestone 6: Release & Deployment

### Deployment Day
**Who's Involved:** Release Manager (lead), Developers, QA, Support Lead, Project Manager

**Deployment Activities:**
- [ ] **Rel Mgr:** Execute deployment according to runbook
- [ ] **Dev:** Monitor deployment progress and logs
- [ ] **QA:** Execute smoke tests in production
- [ ] **Rel Mgr:** Verify all services are healthy
- [ ] **Support Lead:** Monitor support channels for issues
- [ ] **Rel Mgr:** Send release announcement
- [ ] **PM:** Update stakeholders on deployment status
- [ ] **All:** Stand by for immediate post-deployment period

**Post-Deployment:**
- [ ] **Dev:** Monitor production metrics and error rates
- [ ] **QA:** Validate key workflows in production
- [ ] **Support Lead:** Triage any user-reported issues
- [ ] **Rel Mgr:** Complete deployment documentation
- [ ] **PM:** Send deployment completion summary

**Collaboration Checkpoint:**
- [ ] Post-deployment review within 24 hours
- [ ] Document any issues or lessons learned
- [ ] Plan hotfix if critical issues identified

---

## Milestone 7: Post-Release Support

### Support Handoff
**Who's Involved:** Support Lead (lead), Release Manager, Developers, Product Manager

**Handoff Activities:**
- [ ] **Rel Mgr:** Transfer release documentation to Support
- [ ] **Support Lead:** Set up support ticket tracking
- [ ] **Dev:** Be available for escalations and questions
- [ ] **Support Lead:** Document common questions and issues
- [ ] **PdM:** Monitor user feedback and satisfaction
- [ ] **Support Lead:** Escalate bugs and feature requests
- [ ] **PM:** Track post-release issues and resolutions

### Monitoring Phase (First 2 Weeks)
- [ ] **Support Lead:** Daily summary of issues and trends
- [ ] **Dev:** Quick turnaround on critical bugs
- [ ] **PdM:** Review user analytics and adoption metrics
- [ ] **PM:** Coordinate hotfix deployments if needed
- [ ] **Support Lead:** Update FAQ and knowledge base
- [ ] **All:** Collect feedback for retrospective

## Milestone 8: Retrospective & Closure

### Retrospective
**Who's Involved:** All team members

**Preparation:**
- [ ] **PM:** Schedule retrospective meeting
- [ ] **PM:** Share retrospective format and prompts
- [ ] **All:** Reflect on what went well and what to improve
- [ ] **PM:** Gather metrics on delivery and quality

**Retrospective Meeting:**
- [ ] **PM:** Facilitate discussion on successes
- [ ] **PM:** Facilitate discussion on challenges
- [ ] **All:** Identify root causes of issues
- [ ] **All:** Propose action items for improvement
- [ ] **PM:** Prioritize and assign action items
- [ ] **PM:** Document outcomes and share with team

### Project Closure
**Who's Involved:** Project Manager (lead), Product Manager, key stakeholders

**Closure Activities:**
- [ ] **PM:** Document final project metrics vs. goals
- [ ] **PdM:** Analyze success metrics and user impact
- [ ] **PM:** Capture lessons learned document
- [ ] **PM:** Archive project artifacts
- [ ] **PM:** Celebrate team success and recognize contributions
- [ ] **PM:** Share final report with stakeholders
- [ ] **PM:** Add action items to improvement backlog
- [ ] **PM:** Conduct team member feedback sessions

## Continuous Collaboration Best Practices

### Communication Norms
- **Synchronous:** Use for decisions, clarifications, and complex discussions
- **Asynchronous:** Use for updates, documentation, and non-urgent questions
- **Documentation:** Keep decisions and key information in searchable, shared docs

### Meeting Effectiveness
- **Always have an agenda:** Shared before the meeting
- **Timebox discussions:** Respect scheduled time
- **Document decisions:** Capture outcomes and action items
- **Right participants:** Only invite those who need to contribute or decide

### Cross-Role Respect
- **Assume positive intent:** Approach disagreements constructively
- **Value all perspectives:** Each role brings unique expertise
- **Ask questions:** Seek to understand before judging
- **Give credit:** Acknowledge contributions publicly
